# Kibana Dashboard API — Practical Learnings

Findings from building 6 Nutrir dashboards (34 panels total) via the Kibana Saved Objects API on Kibana 8.12.0.

## Critical Missing Fields in Inline Lens Panels

The Kibana visualization docs describe the general structure but miss three fields required for panels to actually render:

### 1. `indexPatternId` inside the layer

Each layer in `datasourceStates.formBased.layers.<layerId>` must include `"indexPatternId": "<data-view-id>"` matching the data view in `references`. Without this, panels hang indefinitely with no error.

### 2. `ignoreGlobalFilters: false` inside the layer

Must be explicitly set in each layer object. Omitting it causes silent render failure.

### 3. `"indexpattern": {"layers": {}}` in datasourceStates

A third sibling alongside `formBased` and `textBased`. All three must be present:

```json
{
  "datasourceStates": {
    "formBased": {
      "layers": {
        "l1": {
          "columnOrder": ["c1", "c2"],
          "columns": { "..." },
          "ignoreGlobalFilters": false,
          "incompleteColumns": {},
          "indexPatternId": "data-view-id",
          "sampling": 1
        }
      }
    },
    "indexpattern": { "layers": {} },
    "textBased": { "layers": {} }
  }
}
```

## Formula Columns Are Not API-Friendly

`"operationType": "formula"` requires hidden sub-columns (suffixed `X0`, `X1`, etc.) with `references` arrays and `tinymathAst` objects that are not auto-generated by the API. These are complex internal structures generated by the Kibana UI.

**Recommendation:** Avoid formulas entirely in API-created dashboards. Use native operations (`average`, `percentile`, `count`, `terms`) instead.

## `match_only_text` Fields Break Silently

Fields mapped as `match_only_text` (e.g., `message`, `error.message` in ECS logs) cannot be used with `last_value` or any operation requiring sorting/aggregation. Instead of showing an error, the panel hangs indefinitely.

Similarly, `wildcard` type fields (e.g., `url.path`) cannot be used with `last_value` — the underlying `top_metrics` aggregation requires segment ordinals which `wildcard` fields don't support. Error: `top_metrics can only collect bytes that have segment ordinals but Field [url.path] of type [wildcard] does not`.

**Always check field mappings before using `last_value`.** Use `terms` as an alternative for both `match_only_text` and `wildcard` fields.

## Ad-Hoc Data Views Don't Work Reliably via API

Inline `adHocDataViews` with `internalReferences` failed to resolve when created via the Saved Objects API (error: "Could not find the data view").

**Always use saved data views** created via the Data Views API, referenced by ID in both the layer's `indexPatternId` and the panel's `references` array.

## Dashboard-Level vs Panel-Level Failures

A single broken panel causes the **entire dashboard** to hang/fail — not just that panel. This makes debugging difficult.

**Iterative panel-by-panel addition** is the only reliable way to identify which panel is broken.

## Top-Level References Must Include Panel Data View References

When the dashboard's top-level `references` array is non-empty (e.g., because it includes tag references), Kibana expects **all** panel data view references to also appear at the top level, prefixed with the panel index. If only tags are present without the panel references, the entire dashboard fails to load silently.

Each inline panel's data view reference must be duplicated at the top level with the format `"name": "<panelIndex>:<original-ref-name>"`.

**Correct:**
```json
{
  "references": [
    { "type": "tag", "id": "my-tag", "name": "tag-ref-my-tag" },
    { "type": "index-pattern", "id": "my-data-view", "name": "p1:indexpattern-datasource-layer-l1" },
    { "type": "index-pattern", "id": "my-data-view", "name": "p2:indexpattern-datasource-layer-l1" }
  ]
}
```

**Wrong (causes dashboard to hang):**
```json
{
  "references": [
    { "type": "tag", "id": "my-tag", "name": "tag-ref-my-tag" }
  ]
}
```

Tag reference names use the convention `"name": "tag-ref-<tag-id>"`, matching what Elastic Agent dashboards use (e.g., `tag-ref-fleet-managed-default`).

**Key insight:** An empty `"references": []` works fine because Kibana falls back to inline panel references. But a non-empty array triggers Kibana to resolve references from the top level, so all panel references must be present there.

## Known Kibana Issues (Related)

The reference inject/extract system for by-value (inline) panels is a known pain point:

- [#126461](https://github.com/elastic/kibana/issues/126461) — inject/extract for by-value panels is incomplete
- [#33825](https://github.com/elastic/kibana/issues/33825) — dashboard references array issues
- [#174497](https://github.com/elastic/kibana/issues/174497) — request for dedicated Dashboard HTTP API for GitOps

The root cause of the tag reference issue is that Kibana's `injectReferences` iterates over ALL top-level references and tries to map them into panels. When the array is non-empty, references not matching any panel (like tags) don't cause problems themselves, but the absence of expected panel references does — Kibana can't find the data view references it needs.

## Debugging Strategy

1. Start with a single known-working panel
2. Add one panel at a time, verify after each addition
3. If the dashboard breaks, the last added panel is the culprit
4. Export a working panel from an existing dashboard (e.g., Elastic Agent) to compare structures
5. Use `GET /api/saved_objects/dashboard/<id>` to inspect the full JSON of working dashboards
6. Compare against a working dashboard's `references` array to understand expected naming conventions
